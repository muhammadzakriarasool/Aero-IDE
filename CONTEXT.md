# Aero IDE - Comprehensive Architecture & Context Report

This document serves as the **definitive memory bank** and architectural blueprint for the **Aero IDE** project. It is intended for all developers, AI agents, and contributors to understand the core logic, technological constraints, and strict anti-patterns governing the codebase.

---

## 1. Executive Summary
Aero IDE is a highly advanced, browser-based integrated development environment powered by Google's Gemini AI and StackBlitz's WebContainer API. It allows users to prompt an AI to generate entire React/Vite web applications, which are then instantly booted, compiled, and previewed directly inside the browser without relying on external remote servers.

---

## 2. Core Technologies Stack
- **Framework**: Next.js 16 (App Router) - Used to serve the IDE interface.
- **Styling**: Tailwind CSS v4 + Shadcn UI components.
- **State Management**: Zustand v5 (leveraging `unstable_ssrSafe` for hydration safety).
- **Virtualization Engine**: `@webcontainer/api` - Boots a complete, isolated Node.js environment directly in the browser tab.
- **Editor**: Monaco Editor (loaded via `next/dynamic` to avoid SSR issues).
- **AI Integration**: Google Gemini (`@google/genai` SDK) specifically configured to output raw JSON mapped to a WebContainer `FileSystemTree`.

## 4. System Architecture Deep-Dive

### 4.1. WebContainer Boot Sequence (`src/features/webcontainer/boot.ts`)
The WebContainer does not run Next.js. It runs a strictly defined **Vite + React** boilerplate.
- When the IDE loads, `bootWebContainer` initializes the container instance.
- It mounts `VITE_REACT_BASELINE`, a foundational `FileSystemTree` containing `package.json`, `vite.config.ts`, `index.html`, and `src/App.tsx`.
- It spawns `npm install` followed by `npm run dev` to start the Vite HMR server.
- The preview URL generated by the container is piped to an `iframe` in the IDE UI.

### 4.2. File System Synchronization & Delta Updates
To prevent the IDE from crashing or losing the user's keystrokes, the sync process avoids destructive global overwrites:
- **Delta Syncing**: `AeroWorkspace.tsx` iterates through the virtual `FileSystemTree`. It uses `fs.mkdir` to create missing directories and `fs.writeFile` to surgically update files.
- **Auto-Save Mechanism**: The Monaco Editor uses a debounced (1000ms) save timer. The user's active keystrokes are stored locally in `draftContent`. Once the debounce resolves, the state is committed to Zustand and written to the WebContainer.

### 4.3. State Management (Zustand & Hydration)
- Defined in `src/lib/store.ts`, the `IDEStore` utilizes `subscribeWithSelector` for granular component re-rendering.
- **Hydration Safety**: React 19's concurrent rendering is mitigated using Zustand's `unstable_ssrSafe` middleware. The state is strictly atomic.
- **No Persistence**: By default, the IDE operates without `localStorage` persistence. The store resets to `INITIAL_IDE_STATE` on page load.

### 4.4. Dependency Management & Concurrency Locking
A major architectural hurdle in browser-based IDEs is managing `node_modules`.
- **The Problem**: WebContainers do not automatically download dependencies when `package.json` changes.
- **The Solution (`checkAndRunInstall`)**: 
  - The IDE intercepts any file sync (from AI or manual saves).
  - It compares the previous `package.json` with the new one.
  - If a change is detected, the IDE programmatically spawns `npm install --no-progress --no-color`.
  - The terminal output is piped directly into the IDE using Web Streams.
  - **Concurrency Lock**: An `isInstalling` boolean in Zustand locks the "Generate" button, "Run/Sync" button, and pauses the Auto-Save watcher to prevent overlapping installations that would corrupt the container.
  - **Safe Reloading**: The dev server iframe is only refreshed if `npm install` exits with code `0`. If it fails, the reload aborts, preventing a White Screen of Death.

## 5. AI Integration Workflow

### 5.1. The API Route (`src/app/api/generate/route.ts`)
- The endpoint leverages `gemini-2.5-flash`.
- **System Prompt Strictness**: The AI is strictly instructed to return a raw, parsable JSON object (using `responseMimeType: 'application/json'`). Markdown ticks (` ``` `) are forbidden.
- The generated JSON perfectly mirrors the `@webcontainer/api` `FileSystemTree` interface.
- It enforces the inclusion of `package.json`, `vite.config.ts`, `index.html`, `src/main.tsx`, and `src/App.tsx`.

## 6. UI / UX Architecture
- **Resizable Panels**: The IDE layout uses `react-resizable-panels` v4. It features a collapsible left pane (Prompt & Settings), a center pane (Monaco Editor & Tabs), and a right pane (Iframe Preview & Terminal).
- **Monaco Configuration**: Monaco is configured to ignore WebContainer semantic TypeScript errors. Because Monaco runs in the host browser, it cannot access the virtual `node_modules` inside the container, so semantic validation is safely disabled (`noSemanticValidation: true`).

---

## 7. Strict Anti-Patterns & Development Rules
> [!WARNING]
> The following rules are non-negotiable. Violating them will fundamentally break the IDE.

1. **NEVER scaffold Next.js inside the WebContainer**: The container's `FileSystemTree` MUST strictly be a Vite + React project. Next.js App Router relies on Node.js features that the browser cannot reliably virtualize.
2. **NEVER mutate the `FileSystemTree` directly**: Always deep clone the tree (e.g., using `JSON.parse(JSON.stringify(tree))`) before updating React state to prevent tearing.
3. **NEVER use numeric pixel values for Resizable Panels sizes**: Always use string percentages (e.g., `defaultSize={"20%"}`) to accommodate `react-resizable-panels` v4 standards and avoid crushed layouts.
4. **NEVER blindly reload the preview iframe**: If `package.json` was modified, you must await `npm install` and verify `process.exit === 0` before triggering `reloadPreview()`.
5. **NEVER statically import Monaco Editor**: Monaco requires browser globals (`window`, `document`). Always use Next.js `dynamic()` with `ssr: false`.
6. **NEVER allow overlapping WebContainer commands**: Always rely on the `isGenerating` and `isInstalling` Zustand locks before spawning new `npm` processes.

## 8. Future Roadmap (Phase 2)
1. **Interactive Terminal Integration**: Replace the current read-only log buffer with `xterm.js` to allow users to manually run CLI commands.
2. **File Explorer**: Replace the flat file dropdown with a nested, interactive file tree component.
3. **State Persistence**: Introduce Zustand `persist` middleware to save the `FileSystemTree` in IndexedDB for session continuity.
4. **Multi-File Context AI**: Extend the Gemini prompt to read all current active files, allowing the AI to mutate existing codebases accurately rather than generating from scratch.
